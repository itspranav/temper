Philosophy

What is Temper?

Most backend applications are state machines. An order moves through Draft → Submitted → Shipped → Delivered. If the state machine is the essential artifact, the surrounding infrastructure follows mechanically from the specification.

Specifications are the source of truth. Everything else is derived. You describe your domain as I/O Automata. The platform verifies correctness through four levels of formal analysis, then generates an event-sourced runtime, OData API, Cedar authorization, and observability — all from the spec.

Use Cases

Two modes, one platform

Same architecture, different deployment shape. What changes is who writes the specs and who sets the policies.

Agent Operating Layer

  • Agents generate verified specs for their plans
  • Cedar default-deny — humans approve expansions
  • Full audit trail with agent identity
  • WASM-sandboxed external access

Application Platform

  • Specs → verification cascade → OData API
  • Event-sourced persistence (Postgres, Turso, Redis)
  • Cedar authorization on every transition
  • Evolution engine captures unmet intents

Pipeline

How it works

Four stages from conversation to production.

01

Describe

IOA specs define states, actions, guards, and invariants.

02

Verify

SMT, model checking, deterministic simulation, property tests.

03

Deploy

Actor runtime, event sourcing, OData API, Cedar auth.

04

Evolve

Production feedback, developer approval, hot-swap.
