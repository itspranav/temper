⚙ How this uses Temper
Entity: Incident

An incident has states Detected → Investigating → Mitigating → Resolved. Events are child entities. The incident can't be resolved until at least one Action event exists (guard).

# incident.ioa.toml
[entity]
name = "Incident"
initial_status = "Detected"

[vars]
has_mitigation = { type = "bool", init = false }

[[actions]]
name = "Investigate"
from = ["Detected"]
to = "Investigating"

[[actions]]
name = "Mitigate"
from = ["Investigating"]
to = "Mitigating"
effects = [{ set = { var = "has_mitigation", value = true } }]

[[actions]]
name = "Resolve"
from = ["Investigating", "Mitigating"]
to = "Resolved"
guards = [{ is_true = "has_mitigation" }]

# Can't resolve without having mitigated first.
# Temper catches this at L1 model checking.
API Flow

# Timeline events are PATCH updates to fields
PATCH /tdata/Incidents('inc-0220-001')
{ "fields": { "events": [...] } }

# Or events could be their own entity set (IncidentEvents)

# State transitions are verified actions:
POST /tdata/Incidents('inc-0220-001')/Temper.Mitigate
POST /tdata/Incidents('inc-0220-001')/Temper.Resolve
Why Temper?

Without Temper, an agent could accidentally "resolve" an incident it never investigated. The state machine makes that impossible — the guard on Resolve requires has_mitigation = true. The trajectory log gives you a full audit trail.
