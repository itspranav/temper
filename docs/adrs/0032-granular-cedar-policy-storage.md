# ADR-0032: Granular Cedar Policy Storage in Turso

- Status: Accepted
- Date: 2026-03-16
- Deciders: Temper core maintainers
- Related:
  - ADR-0030: Hash-gated verification (same hash-gating pattern for content change detection)
  - `crates/temper-store-turso/src/store/policy.rs`
  - `crates/temper-server/src/authz/policy_persistence.rs`
  - `crates/temper-cli/src/serve/bootstrap.rs`

## Context

Cedar policies were managed exclusively in-memory via `ServerState::tenant_policies: Arc<RwLock<BTreeMap<String, String>>>`. Policies written through the management API (`PUT /api/tenants/{tenant}/policies`, `POST /api/tenants/{tenant}/policies/rules`) were lost on server restart.

Decision approvals (`POST /api/tenants/{tenant}/decisions/{id}/approve`) did persist policies to a `tenant_policies` table in Turso — but as a single blob per tenant with no audit metadata, no change detection, and no per-policy provenance. That table is loaded at boot by `recover_cedar_policies`.

Two problems:

1. **Incomplete persistence**: manual policy edits via the API were not durable across restarts.
2. **No audit trail**: no record of which decision generated which policy rule, who approved it, or whether the content changed between writes.

## Decision

### Sub-Decision 1: New `policies` table with per-entry rows

Add a `policies` table alongside the existing `tenant_policies` table:

```sql
CREATE TABLE IF NOT EXISTS policies (
    tenant     TEXT NOT NULL,
    policy_id  TEXT NOT NULL,
    cedar_text TEXT NOT NULL,
    policy_hash TEXT NOT NULL DEFAULT '',
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    created_by TEXT NOT NULL DEFAULT 'system',
    PRIMARY KEY(tenant, policy_id)
);
```

`policy_id` identifies the logical source of a policy entry:
- `"primary"` — manually managed policy blob (from PUT/POST API)
- `"decision:{id}"` — policy generated by approving decision `{id}`

**Why this approach**: A per-entry model enables per-source provenance (which decision generated this rule?) while remaining backward-compatible. The `policy_id` column can accommodate additional categories in the future without schema changes.

### Sub-Decision 2: Hash-gated writes for change detection

`TursoEventStore::save_policy` reads the existing `policy_hash` before writing. If the SHA-256 digest of the new `cedar_text` matches the stored hash, the write is skipped and `Ok(false)` is returned. On a content change (or first insert), an `ON CONFLICT … DO UPDATE` upsert runs and `Ok(true)` is returned.

**Why this approach**: Follows the same pattern as `ADR-0030` (spec content hashing for verification cache), keeping implementation patterns consistent. Avoids unnecessary Turso writes (and trajectory noise) when the same policy is reloaded on repeated restarts.

### Sub-Decision 3: Trajectory entries for policy changes

When `save_policy` returns `true` (content changed), `persist_and_activate_policy` logs a trajectory entry with:
- `entity_type = "_cedar"`, `entity_id = tenant`, `action = "policy_saved"`
- `source = "Platform"`, `agent_id = created_by`

**Why this approach**: Makes policy changes observable in the Evolution Engine dashboard alongside entity action outcomes. Correlating policy approvals with subsequent authorization events aids debugging.

### Sub-Decision 4: Cedar reload is the caller's responsibility

`persist_and_activate_policy` handles only Turso persistence and trajectory logging. Cedar engine reload (`state.authz.reload_policies(...)`) and in-memory map update remain the caller's responsibility. Callers already invoke `validate_and_reload_policies` before calling `persist_and_activate_policy`.

`load_and_activate_tenant_policies` is the exception — it both updates the in-memory map and reloads Cedar, because it is called at boot when there is no prior reload to rely on.

**Why this approach**: Prevents double-reload in the hot path where validation has already reloaded Cedar. The division of responsibility is documented at each call site.

### Sub-Decision 5: Two-pass boot-time recovery

`recover_cedar_policies` in `temper-cli` uses a two-pass strategy:

1. **Legacy pass**: reads from `tenant_policies` (flat blob per tenant) for backward compatibility with data written before this migration.
2. **New pass**: for each tenant in the spec registry, calls `load_and_activate_tenant_policies` which reads from the new `policies` table. New-table data overwrites legacy data for any tenant that has entries.

**Why this approach**: Zero-downtime migration. Deployments with existing `tenant_policies` data continue to work. Once the new table is populated (after the first policy write post-deploy), subsequent restarts use the new table. The legacy pass can be removed after a migration period.

### Sub-Decision 6: Dual-write in `handle_approve_decision` (transition period)

During the transition, `handle_approve_decision` writes to both:
- `tenant_policies` (legacy table, via `upsert_tenant_policy`) — for any consumers still reading the old table
- `policies` (new table, via `persist_and_activate_policy`) — for audit trail and new boot recovery

The legacy write will be removed once the two-pass recovery has been confirmed in production and no other readers of `tenant_policies` remain.

## Rollout Plan

1. **Phase 0 (this PR)** — Add `policies` table, CRUD layer, server helpers, two-pass boot recovery. All API write paths persist to new table. Dual-write in decision approval for backward compat.
2. **Phase 1 (follow-up)** — Add integration tests for `save_policy` / `load_policies_for_tenant` roundtrip. Validate boot recovery in staging.
3. **Phase 2 (cleanup)** — Remove legacy `upsert_tenant_policy` call from `handle_approve_decision` once new table is confirmed as sole reader.

## Consequences

### Positive
- Cedar policies survive server restarts for all write paths (API and decision approvals).
- Per-policy audit trail (who wrote it, when, from which decision).
- Hash-gated writes prevent duplicate Turso traffic on repeat restarts.
- Trajectory entries make policy changes visible in Evolution Engine dashboards.

### Negative
- Two tables (`tenant_policies` + `policies`) exist during the transition period; dual-write adds a Turso round-trip to the decision approval hot path.
- `recover_cedar_policies` is now more complex (two-pass).

### Risks
- Legacy/new table divergence if a write path is missed. Mitigated by the two-pass recovery: new table always wins, legacy fills the gap.
- Hash collision (extremely unlikely with SHA-256) could suppress a legitimate policy update. Mitigated by the `ON CONFLICT … DO UPDATE` upsert which would overwrite if the hash ever matched wrongly.

### DST Compliance
- `persist_and_activate_policy` uses `sim_now()` for the trajectory `created_at` timestamp — determinism preserved.
- Turso SQL layer uses `datetime('now')` for `created_at` in `policies` rows — this is the SQLite server-side clock in the persistence layer (not the actor simulation loop) and is acceptable per existing precedent in all other Turso tables.
- No new `HashMap`/`HashSet` usage; `BTreeMap` used throughout.
- No `tokio::spawn`, `std::thread::spawn`, or multi-threaded patterns introduced.

## Non-Goals

- Migrating existing `tenant_policies` rows to the new `policies` table automatically. Two-pass recovery handles the coexistence period.
- Per-policy RBAC (different principals managing different policy IDs). All policies for a tenant are managed by anyone with `manage_policies` permission.
- Policy versioning / rollback. The new table overwrites on `policy_id` conflict; history is in the trajectory log.

## Alternatives Considered

1. **Extend `tenant_policies` with hash + created_by columns** — Avoids a second table but loses per-policy granularity. Rejected: per-decision audit trail requires a row per decision approval, not one row per tenant.
2. **Store policies as entity events in the event store** — Consistent with the actor model but adds overhead (event replay for policy reconstruction) to a hot boot path. Rejected: the `policies` table is simpler and reads in a single query.
